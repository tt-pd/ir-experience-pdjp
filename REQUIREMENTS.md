# 要件定義

## 目的

PagerDuty Japan の CS / Marketing / Sales チーム向けに、インシデント対応のペインポイントを肌感として体験させる。

- 障害の技術的調査ではなく、**振り分け・エスカレーション・顧客対応**にフォーカス
- 「PagerDutyがない状態」でのインシデント対応がどれだけ辛いかを体験してもらう
- ゲーム / Playbook 形式

> **Why:** CS/Marketing/Sales は顧客やプロスペクトにPagerDutyを説明する立場だが、自分自身がインシデント対応を経験していないためペインポイントが腹落ちしていない。説明の説得力を上げるために、まず自分が体験することが必要。

---

## 対象ユーザー

- PagerDuty Japan の CS / Marketing / Sales 担当者（非エンジニア）
- 個人・自習用（セミナー中の使用も想定）

> **Why:** グループワーク形式も検討したが、ファシリテーターへの依存度が高くスケールしない。個人・自習用にすることで、URLを共有するだけで誰でも体験できるようにした。

---

## ゲーム仕様

| 項目 | 内容 |
|------|------|
| 形式 | ブラウザで動くシナリオ選択ゲーム |
| 想定プレイ時間 | 10分以内 |
| 言語 | 日本語のみ |
| シーン数 | 5シーン |
| 主人公 | CSチームのオンコール当番 |

> **Why（形式）:** Notionのplaybookや紙ベースも検討したが、ファシリテーター依存・スケール不可。Slack Botはホスティングが必要でメンテコストが高い。ブラウザゲームは静的ファイルのみで完結し、URLを渡すだけで体験できるため採用。

> **Why（シーン数5）:** 多すぎると離脱する。少なすぎるとペインが伝わらない。振り分け→エスカレーション→顧客対応→悪化→後処理という一連の流れを最低限カバーできる5シーンで合意。

> **Why（主人公をCS）:** 対象ユーザーが自分ごととして感じやすくするため。エンジニア視点ではなく、CSが実際に直面する「技術的には何もできない状況での判断」にフォーカスした。

### シナリオ設定

| 項目 | 内容 |
|------|------|
| テーマ | 深夜のECサイト決済障害 |
| ゲーム内時刻 | 23:15 〜 翌04:12（約5時間） |
| 前提 | PagerDuty未導入。全選択肢がひどい（相対的にマシなものはある） |

> **Why（深夜のEC決済）:** 影響がわかりやすく、金額換算しやすい。深夜という設定にすることで「担当者が捕まらない」「疲弊感」が演出しやすい。

> **Why（ゲーム内5時間）:** 実際のインシデント対応は長い。プレイ時間は10分以内だが、ゲーム内時刻を5時間進めることで損失額のリアリティを担保する。1兆円設定も検討したが、年商1000億円（中堅〜大手ECの現実的な規模）の方が顧客に説明する際の説得力がある。

> **Why（全選択肢がひどい）:** 正解を当てるクイズにすると「知識ゲー」になってしまい、ペインポイントが伝わらない。PagerDutyがない状態ではどれを選んでも問題が起きるという体験をさせることが目的。ただし相対的なマシ度は残し、「頑張っても限界がある」ことを示す。

### シーン構成

| シーン | ゲーム内時刻 | タイトル | 体験させるペインポイント |
|--------|------------|---------|----------------------|
| Scene 1 | 23:15 | アラート受信 | 誰に連絡すべきかわからない |
| Scene 2 | 23:47 | エンジニアが捕まらない | エスカレーション先が不明確・当番表が古い |
| Scene 3 | 00:23 | 顧客対応が押し寄せる | 障害対応と顧客対応が並走する |
| Scene 4 | 01:55 | 状況悪化 | 追加対応者が誰かわからない・報告フォーマットがない |
| Scene 5 | 03:40 | 収束→後処理 | 障害後の作業量の重さ（選択肢なし・リスト演出） |

> **Why（Scene 5を選択肢なしのリスト演出にした）:** 収束シーンは「やれやれ終わった」で締めると達成感になってしまいペインが薄れる。「直ってからが本番」という感覚を伝えるため、選択肢ではなくタスクが次々と降ってくるリスト演出にした。

> **Why（Scene 4は分岐なし・1本）:** 分岐を増やすとシナリオの管理コストが上がる。ミニマム実装を優先し、どのScene 2の選択肢を選んでもScene 3以降は同じフローにした。

### 選択肢の設計

- 各シーン3択
- 全選択肢がひどい前提。`bad` / `okay` / `best` の3段階で相対評価
- 選択後にフィードバックテキスト（bad/okay/bestラベル＋理由）を表示
- フィードバック直下にそのシーン中の推定損失額を表示

> **Why（選択後に即フィードバック）:** フィードバックは早い方が学習効果が高い。エンディングまで引っ張ると「なぜそれが問題だったか」が薄れる。シーンごとに損失額を出すことで、時間が経つほど金額が積み上がるリアリティも加わる。

## タイマー・損失額

| 項目 | 内容 |
|------|------|
| 実経過タイマー | ヘッダーに常時表示（MM:SS） |
| プログレスバー | 600秒（10分）を満タンとして表示、超過後は色変化 |
| タイムアウト | 600秒経過でバナー表示「10分が経過しました」、続行可能 |
| ゲーム内時刻 | ヘッダーに常時表示（HH:MM） |
| 損失額（シーン中） | 選択後フィードバックに表示（ゲーム内経過分 × 秒単価） |
| 損失額（エンディング） | ゲーム内約5時間ベースの総額をカウントアップ表示 |
| 年商設定 | 1000億円（3,171円/秒） |
| 表示フォーマット | 日本語桁区切り（億・万円単位） |

> **Why（タイマーは実経過のみ表示、損失額はエンディングまで非表示）:** プレイ中に損失額を常時出すと数字に慣れてしまい、エンディングのインパクトが薄れる。時間プレッシャーはタイマーだけで十分に与えられる。損失額はシーン終了後とエンディングで出すことで「気づき」として機能させる。

> **Why（タイムアウト後も続行可能）:** 強制終了にするとプレイヤーが全シーンを体験できなくなる。「10分を超えた」という事実を突きつけつつ、体験は完走させる設計にした。

> **Why（年商1000億円）:** 最初は1兆円を検討したが、大きすぎてリアリティがなくなる。1000億円は中堅〜大手ECとして現実的で、かつ秒単価3,171円という数字が「たった1秒でも」と感じさせる絶妙なラインと判断した。

> **Why（日本語桁区切り）:** 数字のみの表示（¥56,520,900）は桁が多すぎて瞬時に読めない。「約5,652万円」の方が非エンジニアに刺さる。

## エンディング

1. 「障害は直りました。でも、あなたの夜はまだ終わっていません。」
2. 推定総損失額をカウントアップ表示（約5,652万円）
3. 後処理リストが時間差で順番に出現
   - 影響顧客への個別連絡 → 対象者リストがない
   - ステークホルダーへの事後報告 → フォーマットがない
   - 再発防止ドキュメント → 誰が書くか決まっていない
   - 問い合わせ収束確認 → まだ来ている
   - 次回当番への申し送り → 仕組みがない

> **Why（後処理リストを時間差で出す）:** 一度に全部見せると読み飛ばされる。1つずつ出てくることで「まだある、まだある」という疲弊感を演出する。

> **Why（クビやメディア炎上などの演出を省いた）:** 過剰な演出はゲーム感が強くなりすぎて、リアルなペインポイントの訴求から外れる。金額だけの方がシンプルで説得力がある。

## 技術スタック

| 項目 | 内容 |
|------|------|
| HTML | `index.html` 1ファイル（UI + ゲームエンジン） |
| JS | Vanilla JS（ビルドなし） |
| CSS | Tailwind CSS CDN |
| シナリオ | `scenario.js`（JSオブジェクト、外部ファイルとして分離） |
| デプロイ | GitHub Pages / Vercel（静的、メンテ不要） |
| フォント | Noto Sans JP（Google Fonts CDN） |

> **Why（Vanilla JS + Tailwind CDN）:** React等のフレームワークはビルド環境が必要でメンテコストが上がる。このスケールのゲームであればVanilla JSで十分。Tailwind CDNにすることでCSSのビルドも不要。

> **Why（scenario.jsを分離）:** index.htmlにシナリオを埋め込むと、シナリオを追加・修正するたびにゲームエンジン部分に触れることになる。シナリオ担当（CS/Marketing）がエンジニアなしで編集できるように分離した。

> **Why（静的ファイルのみ）:** サーバーが必要な構成はホスティングコストとメンテが発生する。静的ファイルであればGitHub Pagesに無料でデプロイでき、URLを渡すだけで体験できる。
